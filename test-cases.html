1. Resumo Executivo

Esta rodada de testes validou o fluxo completo de provisionamento e a experiência de checkout do aplicativo de doação: leitura e reinicialização do estado do sistema, criação do catálogo de produtos, mecanismos de proteção contra operações fora de ordem, configuração de imagem e preço do SKU, provisionamento de estoque nos warehouses disponíveis — concluindo com a transição para o estado READY — e por fim a validação do componente DonationMinicart no carrinho da loja (FASE-06, appStatus=ACTIVE). Dos 28 casos de teste executados, 27 foram aprovados e 1 foi intencionalmente pulado por não ser reproduzível no ambiente de testes — nenhum caso foi reprovado. Durante a sessão foram identificados dois achados relevantes: um defeito corrigido (TC-05.1) e um comportamento sem proteção de idempotência observado na API de configuração (TC-03.6).

🔍

2. O que foi testado

  • ✓ Consulta do estado do sistema: o sistema consegue informar, a qualquer momento, em qual etapa de configuração se encontra.
  • ✓ Reinicialização do sistema: o sistema restaura o estado padrão corretamente quando acionado pelo administrador.
  • ✓ Criação do catálogo: o sistema cria adequadamente o produto e o item (SKU) de doação no catálogo da loja, registrando seus identificadores para uso nas etapas seguintes.
  • ✓ Proteção contra execução fora de ordem: o sistema recusa e notifica o administrador quando uma etapa é acionada antes que seus pré-requisitos sejam atendidos (ex: tentar configurar estoque antes de criar o catálogo).
  • ✓ Proteção contra reprocessamento desnecessário: o sistema não recria o catálogo se ele já houver sido criado, evitando duplicações de produto na loja.
  • ✓ Validação de pré-requisitos de imagem: o sistema bloqueia o provisionamento de estoque quando a imagem do produto ainda não foi configurada.
  • ✓ Reaproveitamento de cadastros existentes: ao executar a criação do catálogo com produto e SKU já presentes na loja, o sistema reconhece os registros existentes e os reutiliza — sem criar itens duplicados.
  • ✓ Validação de valores de doação: o sistema recusa valores de doação fora da lista de opções autorizadas (R$ 1, 2, 3, 5, 6, 10, 15), independentemente do formato enviado (número inválido, zero, ou texto).
  • ✓ Configuração de imagem e preço do SKU: o sistema configura a imagem e o preço do item de doação em paralelo com sucesso, liberando a etapa seguinte de provisionamento de estoque.
  • ✓ Comportamento de reconfiguração: o sistema aceita uma segunda chamada de configuração para o mesmo SKU já configurado, atualizando o preço sem rejeitar a operação.
  • ✓ Listagem de warehouses disponíveis: o sistema retorna a lista de depósitos de estoque disponíveis no ambiente, com identificadores e nomes.
  • ✓ Validação do payload de warehouses: o sistema rejeita tentativas de provisionamento de estoque com lista vazia, campo ausente ou valor em formato incorreto (string em vez de array).
  • ✓ Provisionamento de estoque nos warehouses: o sistema configura estoque infinito nos depósitos selecionados e avança para o estado READY, registrando os identificadores dos warehouses utilizados.
  • ✓ Proteção contra reprovisionamento: o sistema rejeita uma segunda tentativa de provisionamento de estoque com HTTP 409, mantendo o estado READY intacto.
  • ✓ Ativação do aplicativo no checkout: o aplicativo de doação foi habilitado pelo operador para o estado ACTIVE, tornando o componente visível no minicart.
  • ✓ Componente de doação no minicart: o DonationMinicart renderizou corretamente na gaveta do carrinho, exibindo o checkbox desmarcado e o rótulo de doação — sem erros de console.
  • ✓ Adição do produto de doação: ao marcar o checkbox, o SKU 115 foi adicionado ao orderForm com quantity=1, conforme esperado.
  • ✓ Remoção do produto de doação: ao desmarcar o checkbox, o SKU 115 foi removido do orderForm — sem deixar rastros no carrinho.
  • ✓ Resiliência com carrinho vazio: ao esvaziar o carrinho completamente, o componente não gerou nenhum erro JavaScript — comportamento esperado de plataforma VTEX.

5. Achados relevantes

⚠️ Defeito identificado e corrigido — TC-05.1

Durante o teste de reaproveitamento de cadastros existentes, o sistema retornou erro interno ao tentar localizar um produto e um SKU já cadastrados. A investigação revelou que os endereços de consulta usados internamente estavam incorretos — o sistema buscava os registros em rotas da API do catálogo que não correspondem à documentação oficial da plataforma VTEX.

Impacto do defeito: em qualquer reexecução do provisionamento de catálogo, a aplicação falharia com erro interno em vez de reconhecer os itens já existentes. Isso poderia também causar tentativas de criação de produtos duplicados na loja.

✅ Correção aplicada

Os endereços de consulta foram ajustados para as rotas corretas da API oficial do VTEX Catalog. Após a correção e republicação do aplicativo, o TC-17.2 foi reexecutado com sucesso, confirmando que o sistema agora reconhece corretamente os produtos já cadastrados.

📐 Comportamento observado — TC-03.6: ausência de proteção contra reconfiguração

A API configureProduct aceita novas chamadas enquanto o status for CATALOG_CREATED, sem verificar se a configuração já foi concluída. Ao ser chamada pela segunda vez com price=5, o sistema reprocessou normalmente e atualizou o preço de R$ 3 para R$ 5.

Análise: o comportamento pode ser intencional (permite ao administrador alterar o valor de doação antes de provisionar o estoque), mas não está documentado como funcionalidade explícita. Não classificado como defeito. O preço vigente no sistema é R$ 5.

Recomendação: documentar se a reconfiguração antes do provisionamento de estoque é um caso de uso suportado e considerar log de auditoria para rastrear alterações de preço.
